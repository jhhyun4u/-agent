# 용역 제안 성공 지식 베이스
> 시스템 프롬프트 및 AI 에이전트 설계 기반 지식

---

## 1. 성공의 핵심 원칙 (5가지)

| 원칙 | 설명 |
|------|------|
| 발주처 중심 사고 | 우리 능력 자랑이 아닌 발주처 문제 해결에 집중 |
| 명확한 차별화 | "왜 우리인가?" — Win Theme으로 전체 제안서 관통 |
| 증거 기반 주장 | 추상적 약속 ❌ / 수치·실적·사례 ✅ |
| 규칙 준수 | 형식 오류 = 자동 탈락. RFP 규격은 절대 준수 |
| 팀 협력·품질관리 | 스토리보드로 통일, Color Team Review로 다각도 검증 |

---

## 2. 핵심 전략 프레임워크

### 2-1. Win Theme 방정식
```
Win Theme = 발주처 Hot Button × 우리 강점 × 경쟁사 약점
```
- 제안서 표지~본문 전체에 일관되게 반복
- 3개 이내 핵심 메시지로 압축

### 2-2. Ghost / Win Theme / Action Forcing Event
| 개념 | 정의 | 예시 |
|------|------|------|
| Ghost Theme | 발주처가 직면한 근본 문제 | "노후 시스템으로 인한 업무 지연" |
| Win Theme | 우리만의 해결책 차별점 | "AI 자동화로 처리량 3배 증대" |
| Action Forcing Event | 지금 결정해야 하는 이유 | "정책 시행 일정 내 전환 필수" |

### 2-3. Shipley Color Team Review 시스템
| 팀 | 시점 | 역할 |
|----|------|------|
| Blue Team | RFP 발행 전/초기 | 전략 수립 검증 |
| Pink Team | 1차 초안 완성 | 전략 반영도 검수 |
| Green Team | 2차 초안 | 가격·원가 검증 |
| Red Team | 최종 초안 | 경쟁 관점 전체 검수 |
| Gold/Compliance Team | 제출 전 | 규격·형식 최종 확인 |

---

## 3. RFP 분석 체크리스트

| 항목 | 목적 |
|------|------|
| 사업 배경·목적 | 발주처의 진짜 고민 파악 |
| 평가 배점표 | 배점 역설계 → 점수 집중 영역 결정 |
| 필수 요건 (Mandatory) | 미충족 = 실격 → 100% 충족 확인 |
| 작성 서식·분량 제한 | 형식 미준수 = 감점·탈락 |
| 기술:가격 비중 | 전략 방향 결정 (기술 집중 vs 가격 집중) |
| 특이 조건 | 지역업체·중소기업 가산점, 서식 템플릿 등 |

### 배점 역설계 원칙
- 기술 60:가격 40 → 기술 차별화 중심
- 기술 40:가격 60 → 가격 경쟁력 중심
- 배점 높은 항목 = 발주처 최우선 관심사 → 분량·깊이 집중

## 4. 제안서 필수 섹션 및 스토리라인

### 4-1. 표준 섹션 구성
1. 사업 개요·배경 (As-Is → To-Be 문제 제기)
2. 수행 방안·기술 접근 (우리만의 방법론)
3. 조직·인력 계획 (핵심 인력 전문성)
4. 예산·일정 (현실적·구체적 근거)

### 4-2. 스토리라인 구성 4단계
```
Step 1: 발주처 니즈 정의 (As-Is / To-Be / 핵심 과제 3~5개)
   ↓
Step 2: 우리 솔루션 제시 (차별화 접근법 + 기대 효과)
   ↓
Step 3: 실행 가능성 입증 (실적 + 인력 + 유사 사례)
   ↓
Step 4: 리스크 관리 + 부가가치 제시
```

### 4-3. 작성 원칙
- **발주처 언어 사용**: RFP 용어 그대로 → "우리가 이해하고 있다"는 신호
- **1슬라이드 1메시지** (PPT)
- **수치화**: "최신 기술 적용" ❌ → "○○ 기술 적용, 처리시간 30% 단축" ✅
- **도식화 우선**: 평가위원은 그림을 더 빠르게 신뢰

---

## 5. 입찰가격 전략

### 가격-기술 트레이드오프 시뮬레이션
```
최종점수 = (기술점수 × 기술배점%) + (가격점수 × 가격배점%)
→ 기술점수 강화 비용 vs 가격 인하 효과 중 어느 쪽 ROI가 큰가?
```

### 가격 전략 원칙
- G2B 낙찰 사례(동종 유사 공고)로 합리적 가격 밴드 파악
- 낙찰률 통계 기반 예정가격 추정
- 너무 낮으면 이행능력 의심 → 적격심사 탈락 가능
- 기술점수 1위면 가격을 올려도 낙찰 가능한지 계산

## 6. 실패 패인 분석 (절대 하지 말 것)

| 패인 | 원인 |
|------|------|
| 형식 탈락 | 규격·분량·서명 오류 |
| RFP 무시 | 발주처 요구와 무관한 자기 자랑 |
| 전략 없는 나열 | Win Theme 부재, 단순 기술 목록 |
| 섹션 간 불일치 | 파트별 따로 작성 후 취합 → 논리 단절 |
| 가격 오판 | 경쟁사 동향·낙찰률 미분석 |
| 신뢰 부족 | 유사 수행 실적 없음 |

## 7. 한국 공공조달 특수성

### 나라장터 제출 전 필수 확인
- [ ] RFP 목차 100% 준수
- [ ] 필수 첨부서류 전부 포함
- [ ] 분량 지정 범위 내 (초과·미만 모두 탈락 가능)
- [ ] 표지·폰트·용지 규격 정확
- [ ] 법인대표 서명·도장
- [ ] 나라장터 제출 후 재다운로드 확인

### 평가위원 특성
- 6년 이상 해당 분야 경력 또는 기술사·박사
- 기술적 깊이 이해 가능 → 전문용어 남발보다 논리가 중요
- 여러 제안서를 짧은 시간에 검토 → **첫 3페이지에 핵심 메시지**

## 8. AI 에이전트 설계 시 활용 포인트

### 제안전략 AI 에이전트 필수 기능
1. **배점표 역설계 엔진**: RFP에서 배점 자동 추출 → 집중 영역 추천
2. **Hot Button 탐지기**: 반복 키워드·강조 표현 자동 파악
3. **Win Theme 생성기**: 발주처 니즈 × 자사 강점 × 경쟁사 약점 분석
4. **Compliance Matrix**: 요구사항 vs 제안서 매핑 자동화
5. **스토리라인 생성**: 4단계 스토리 구조에 맞는 섹션별 방향 제시
6. **자가진단 평가**: 작성된 제안서를 평가위원 관점에서 채점

### 병렬 처리 가능 작업
- 4개 섹션 (사업개요/수행방안/조직인력/예산일정) 동시 작성
- 제안계획 수립 세부 작업들 (팀구성, 일정, 스토리라인, 가격추천) 동시 진행
- PPT 각 섹션 슬라이드 동시 생성

### Human 개입 필수 게이트
- 제안전략 승인 (전략의 방향성은 사람이 판단)
- 가격 최종 결정 (민감한 비즈니스 판단)
- 최종 제안서 리뷰 (법적 책임 주체는 사람)
