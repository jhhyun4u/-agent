> v3.3 → v3.4: ProposalForge 프론트엔드 비교 검토 + API 갭 해소 — 표준 에러 코드 체계(§12-0), 섹션 AI 재생성/인라인 어시스턴트/모의평가 조회/버전 Diff/HWP 내보내기 API 5건(§12-4 확장), 노임단가/낙찰가 CRUD(§12-12 확장), 분석 대시보드 집계 4건(§12-13 신규), 프론트엔드 컴포넌트 13건(§13, §31). 기존 부분 반영 2건 품질 향상.
> 아키텍처 결정: Pattern A(모놀리식 StateGraph + Send() 병렬처리) 유지 확정.
> 잔여 미반영: HIGH 6건 (AUTH-06, PSM-05, PSM-16, POST-06, OPS-02, OPS-03) + 부분 반영 8건 — 구현 단계에서 점진 보완.

### v3.3 점수 상승 근거

- **핵심 워크플로 98% → 99%**: self_review 5방향 원인별 피드백 라우팅(retry_research/retry_strategy/retry_sections) 확장으로 품질 게이트 정교화. plan_price에 labor_rates/market_price_data 실데이터 조회 로직 추가로 "[단가]" 플레이스홀더 해소. review_plan → strategy_generate 역류 분기(전략-예산 상호조정) 추가.
- **TRS 95% → 98%**: TRS-09(AI 생성 vs Human 편집 구분) — artifacts 테이블에 `change_source` 컬럼 추가('ai_generated'|'human_edited'|'quality_gate'|'final')로 직접 해결. 기존 HIGH 미반영 → 완전 반영.
- **AI 모니터링 92% → 95%**: §22-4 Fallback 전략 추가(Claude API 재시도, 외부 데이터 graceful degradation, 품질 게이트 반복 실패 한도, 노드별 타임아웃)로 AGT 영역 운영 안정성 강화.
- **비기능/운영 90% → 93%**: §22-4 Fallback 전략이 OPS-04~06을 구체 코드 레벨로 설계(기존 §28-8은 정책 수준). labor_rates/market_price_data Fallback(직전 연도, 데이터 스킵+태그) 포함.
- **전체 96% → 97%**: TRS-09 HIGH 해소(1건) + 워크플로/운영 품질 상승분 반영.

### v3.4 점수 변동 근거

- **KB 95% → 96%**: 노임단가(labor_rates) CRUD API(§12-12 H4)와 낙찰가 벤치마크(market_prices) CRUD API(§12-12 H5)가 §15-5h/5i 데이터 모델에 대응하는 완전한 REST 엔드포인트를 갖춤. v3.3에서 DB 스키마만 있고 API가 없던 갭 해소. 프론트엔드(§13-13 LaborRatesTable/MarketPricesTable)와 정합.
- **TRS 98% → 99%**: §12-4-1 섹션 AI 재생성 API에서 `change_source: "ai_revised"` 반환, §12-4-4 버전 Diff API에서 임의 버전 간 교차 비교 지원 → TRS-09(AI/Human 편집 추적)과 ART-03(버전 비교)의 반영 품질 한 단계 향상.
- **비기능/운영 93% → 94%**: §12-0 표준 에러 코드 체계(8 프리픽스, 15 에러 코드)가 NFR-09(오류 처리)의 프론트엔드 핸들링 품질을 높임. 에러 종류별 자동 재시도/토큰 갱신/인라인 에러 구분이 §22-4 Fallback 전략과 일관성 있게 체계화.
- **프론트엔드-백엔드 정합성 98%**: v3.4에서 추가한 프론트엔드 컴포넌트(§13-10~13) 13건에 대응하는 백엔드 API가 모두 설계됨. H1~H6(프론트엔드 필수) 6건 + M1~M3(품질 개선) 3건 = 9건 API 갭 해소. L1~L2(향후) 2건만 잔여.
- **전체 97% 유지**: v3.4 변경은 기존 완전 반영 항목의 품질 향상(API-프론트엔드 정합)이 주이며, HIGH 미반영 6건은 변동 없음. 소수점 단위 상승(~97.3%)이나 반올림 기준으로 97% 유지. 부분 반영 8건은 변동 없음 — 구현 단계에서 점진 보완.

### 6a-4. v3.4 핵심 시사점

1. **HIGH 갭 변동 없음**: v3.4 변경은 프론트엔드-백엔드 API 정합성 확보가 핵심 목적이며, 잔여 HIGH 6건(AUTH-06, PSM-05, PSM-16, POST-06, OPS-02, OPS-03)은 모두 백엔드 아키텍처/운영 설계 영역으로 v3.4 범위와 무관.
2. **프론트엔드 구현 준비 완료**: v3.4에서 추가한 프론트엔드 컴포넌트 13건에 대응하는 백엔드 API가 모두 설계됨. H1~H6(필수 6건) + M1~M3(품질 3건) = 9건 갭 해소로 프론트엔드 구현 시 API 부재로 인한 블로커 없음.
3. **표준 에러 코드의 파급 효과**: §12-0의 에러 코드 체계는 ProposalEditor 자동저장 시나리오뿐 아니라, 전체 API의 에러 핸들링 일관성을 높임. 구현 시 `TenopAPIError` 기반 클래스를 점진적으로 적용하면 NFR-09 요구사항의 완성도가 더 올라감.
4. **HWP 내보내기 추가**: 요구사항에 명시되지 않았으나 한국 공공조달 현실에서 필수. 요구사항 갱신 시 ART 항목에 반영 권고.
5. **NEW 갭 없음**: v3.4에서 도입된 신규 기능(HWP, 인라인 어시스턴트 등)은 모두 기존 요구사항의 확장이거나 설계 자체적 품질 향상이며, 요구사항과 충돌하는 항목 없음.
