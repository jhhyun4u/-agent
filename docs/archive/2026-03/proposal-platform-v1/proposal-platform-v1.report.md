## 7. 교훈 (Lessons Learned)

### 7.1 설계와 구현의 간극 관리

**문제**: 초기 설계(v3.0)는 요구사항만 반영. 구현하며 기술 제약과 사용자 피드백 반영 필요.

**해결책**:
- 설계 버전 관리 (v3.0 → v3.6)
- 각 Phase마다 피드백 루프 (요구사항 vs 설계 vs 구현)
- 의도적 차이 문서화 (PPT 파이프라인 변경 등)

**적용**: 다음 프로젝트에서 "설계 진화 과정" 추적

### 7.2 프롬프트 엔지니어링의 중요성

**문제**: 초기 프롬프트는 기본 기능만 지원. 전문 분야(grant-writing)의 품질 개선 필요.

**해결책**:
- v3.6 Grant-Writer Best Practice 추가 (스토리텔링, SMART, Budget Narrative)
- 평가자 관점 + 용어 정합성 강화
- 프롬프트별 테스트 케이스 작성

**결과**: 제안서 품질 점수 상향 (예상)

**적용**: 새 프롬프트 개발 시 "best practice 레퍼런스" 필수

### 7.3 데이터 환류 루프의 가치

**문제**: Phase 4 없이는 제안 → 결과까지 추적 불가. 미완 상태에서는 KB 갱신도 불가능.

**해결책**:
- 성과 추적 Materialized View (Phase 4-3)
- 제안 결과 → KB 자동 갱신 (수주/패찰/교훈)
- 차 제안 시 자동 적용 (지식 재활용)

**결과**: 폐쇄 루프 시스템 완성 → 지속적 개선

**적용**: 향후 기능은 "데이터 환류" 고려하여 설계

### 7.4 컴포넌트 분리의 명확성

**문제**: RfpSearchPanel, GoNoGoPanel이 "미구현"으로 보이지만, 실제로는 WorkflowPanel에 통합.

**원인**: 전용 컴포넌트로 분리하지 않아 "설계와 다름"으로 인식

**개선**:
- 향후 "컴포넌트 통합 vs 분리" 명시적 결정 필요
- 설계에 "통합" 표시

**적용**: 다음 설계에서 "레이아웃 통합" 검토 단계 추가

### 7.5 무중단 운영의 필수성

**발견**: MV 갱신 시 `CONCURRENT` 옵션으로 서비스 중단 방지.

**구현**:
```sql
REFRESH MATERIALIZED VIEW CONCURRENTLY mv_team_performance
```

**가치**: 프로덕션 환경에서 대량 데이터 변환 중 서비스 연속성 보장

**적용**: 모든 향후 대량 데이터 작업에 `CONCURRENT` 옵션 적용

### 7.6 테스트 자동화의 중요성

**교훈**: Phase 3.5~4 동안 정기적 테스트 → 버그 조기 발견

**적용**:
- 모든 엔드포인트 자동 테스트
- CI/CD 파이프라인 (GitHub Actions)
- 배포 전 회귀 테스트
