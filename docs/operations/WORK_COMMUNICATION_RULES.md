# 작업 제안 커뮤니케이션 규칙

**목적:** Phase/STEP 라벨이 아닌 "기능 설명"으로 작업을 명확하게 전달  
**작성일:** 2026-04-21

---

## 핵심 규칙

**모든 작업 제안 시, 반드시 이 순서를 따를 것:**

```
1️⃣ 기능 설명 (WHAT)
   └─ 사용자 관점에서 이 기능이 뭘 하는지?
   └─ 어떤 문제를 해결하는가?

2️⃣ Phase/STEP 라벨 (WHERE IN ROADMAP)
   └─ Phase 5, STEP 8 등 내부 단계

3️⃣ 기술 세부사항 (HOW)
   └─ 구현 기술, 소요 시간, 우선순위
```

## ❌ 나쁜 예 (기존 방식)

"STEP 4A Accuracy 개선"

## ✅ 좋은 예 (개선된 방식)

### 1. 정기 문서 마이그레이션 스케줄러 (Phase 5)

사용자 입장:
- 📅 지정된 시간에 자동으로 인트라넷 문서 마이그레이션 실행
- 🔄 최대 100개 문서를 5개 워커로 병렬 처리 (~30초)
- 📊 각 배치별 처리 상태 추적 및 조회 가능
- 🔔 Teams Webhook으로 완료 알림

기술:
- APScheduler 기반 정기 실행
- ThreadPoolExecutor 병렬 처리
- 6개 REST API 엔드포인트
- 배포 상태: 스테이징 완료 (2026-04-21), 프로덕션 2026-04-25 예정

---

### 2. 비동기 Job Queue 시스템 (STEP 8)

사용자 입장:
- ⚡ 긴 작업(정확도 검증, 섹션 재작성)을 백그라운드에서 처리
- 📤 사용자는 즉시 응답 받고 나중에 결과 확인 가능
- 🔗 실시간 상태 업데이트 (WebSocket)
- 📋 작업 큐 관리 및 우선순위 조정

기술:
- Celery 또는 RQ 기반 Job Queue
- Redis 메시지 브로커
- 우선순위: HIGH
- 소요 시간: 2-3일

---

### 3. 제안서 섹션 진단 정확도 향상 (STEP 4A)

사용자 입장:
- 🎯 AI가 생성한 각 섹션의 품질을 자동 평가
- 📊 신뢰도/정확도 점수로 의심 섹션 자동 표시
- ✅ 95% 이상의 높은 정확도 달성 목표
- 🔄 앙상블 투표로 여러 변형 섹션 중 최적 선택

기술:
- 신뢰도 추정 (variance-based)
- 앙상블 투표 (weighted averaging)
- k-fold 교차 검증
- 우선순위: MEDIUM
- 소요 시간: 1-2일
