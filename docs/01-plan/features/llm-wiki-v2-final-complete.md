## 1. Knowledge Base 8개 Category (최종 확정)

### ✅ **Category 1: 인력 자산 & 경력기술서 기초 자료**

**정의**: 경영기획실 배포 "인력정보 주기적 배포 파일" + 서식 샘플

**구조**:
```
경영기획실 배포 (분기별):
  ├─ 간략한 이력서 (직급, 전공, 경력연수, 주요경력)
  ├─ 참여과제 목록 (과제명, 역할: 책임자/참여연구원)
  └─ 인력 프로필 데이터

KB 저장:
  ├─ 경영기획실 배포 파일 (원본 보관)
  ├─ 경력기술서 서식 샘플 (정부/민간/국방)
  └─ 메타: 배포일, 버전, 적용 기간
```

**사용 방식**:
```
제안팀이 경영기획실 배포 파일 참고
  → "이 과제에 누가 참여할까?"
  → KB에서 인력 기본정보 + 서식 샘플 활용
  → 서식에 맞게 수작업으로 경력기술서 생성

KB의 역할:
  - 배포파일 중앙 보관 (버전 관리)
  - 서식 샘플 제공 (참고용)
  - 인력 기본정보 저장 (프로필)
```

**업데이트**:
- 분기별 (경영기획실 배포 시마다)

### ✅ **Category 2: 수행실적 + 실적증명서/계약서 보관 (중앙화)**

**정의**: 과제별 실적증명서 또는 계약서 원본을 KB에 파일 업로드 + 메타 관리

**구조**:
```
메타 DB (Supabase):
  프로젝트명 | 발주처 | 계약금액 | 수행기간 | 담당팀 | 주요성과 | 수행일자 | 
  실적증명서파일(링크) | 계약서파일(링크) | 문서타입 | 최신화일

파일 저장소 (Supabase Storage):
  /project-documents/{프로젝트ID}/
    ├─ proof-of-performance.pdf (발주처 발급 실적증명서)
    ├─ contract.pdf (계약서 원본)
    └─ contract-copy.pdf (사본)
```

**사용 방식**:
```
제안팀이 실적 필요 → 실적 목록 검색 (금액, 분야, 발주처 필터링)
  → "의료AI, 50억 규모 프로젝트" 자동 추천
  → 해당 프로젝트의 실적증명서/계약서 링크 클릭
  → 파일 다운로드 → 제안서에 첨부

장점:
  - 중앙화된 저장소 (개인 PC 폴더 대신)
  - 버전 관리 (원본/사본 구분)
  - 검색 가능 (금액, 분야, 발주처별)
  - 재사용 (매 제안마다 활용)
```

**업데이트**:
- 프로젝트 완료 후 즉시 (PM)
- 파일 경로는 불변 (업데이트 안 함)

### ✅ **Category 3: 제출서류 템플릿 (서식 라이브러리)**

**정의**: 제출 시 필요한 각종 서식 템플릿 & 가이드

**구조**:
```
A. 법률/행정 서식
  ├─ 비밀유지서약서 (일반/상대비밀/국방)
  ├─ 참여현황서
  ├─ 자격요건 확인서
  └─ 기타 정부 규정 서식

B. 공고유형별 서식
  ├─ 정부과제 (나라장터 표준)
  ├─ 국방과제
  ├─ 연구용역
  └─ 민간과제

C. 체크리스트
  ├─ 제출 전 필수 확인사항
  └─ 공고별 필수 제출 서식 체크리스트
```

**사용 방식**:
```
RFP 업로드 → 공고유형 자동 분류
  → 필요 서식 체크리스트 자동 생성
  → 템플릿 제공 (구체적 작성 방법 포함)
```

**업데이트**:
- 반기별 + 정부 규정 변경 시 즉시

### ✅ **Category 4: Q&A 아카이브 & 발표 시사점**

**정의**: 과거 발표에서의 평가위원 질의 + 우리 답변 + 평가결과

**구조**:
```
메타 DB:
  분류 | 발주처 | 프로젝트명 | 질문 | 우리답변 | 평가위원반응 | 
  낙찰여부 | 기술점수 | 발표일 | 기록자
```

**발표 시 활용**:
```
"국방부 AI 이상탐지" 발표 예정
  ↓
KB 검색 (발주처 + 분야)
  ↓
과거 Q&A 5건 + 시사점:
  - "보안 질문 자주 나옴 → 기술점수 차이 큼"
  - "우리 답변 방식 OOO가 점수 높음"
  
  ↓
발표 대비 강화
```

**업데이트**:
- 발표 직후 (질문 기록)
- 결과 공시 후 (평가점수 함께)

### ✅ **Category 5: 회사정보 & 차별성 자료**

**정의**: 제안서 작성 시 회사 강점을 다르게 부각시킬 수 있는 기초자료

**구조**:
```
A. 기본 회사정보
  ├─ 회사 소개 (역사, 비전, 규모)
  ├─ 조직도 & 인원 현황
  ├─ 사업 실적 (매출, 성장률)
  └─ 인증/자격 (ISO, 특허, 수상)

B. 차별성 부각 자료 (분야별)
  ├─ IT/클라우드 분야 강점 사례
  ├─ 의료/헬스케어 분야 강점 사례
  ├─ 국방/보안 분야 강점 사례
  └─ ... (분야별)

C. 고객별 강점 맵
  ├─ 삼성과제 → "대규모 프로젝트 경험" 강조
  ├─ 국방부과제 → "보안, 신뢰성" 강조
  ├─ 정부과제 → "컴플라이언스" 강조
  └─ ... (고객별)

D. 시각화 자료
  ├─ 회사 성장 그래프
  ├─ 분야별 매출 비중
  ├─ 주요 프로젝트 포트폴리오
  └─ 팀별 전문성 인포그래픽
```

**사용 방식**:
```
제안서 "회사소개" 섹션 작성
  ↓
RFP 분석: "이 과제의 발주처와 분야는?"
  ↓
KB에서 해당 분야/고객의 차별성 자료 선택
  ↓
"기술력" 사례 또는 "신뢰성" 사례 활용
  ↓
차별성 부각
```

**업데이트**:
- 월별 (회사 정보 변경 시)
- 분기별 (자료 신규 추가)

### ✅ **Category 6: 가이드라인 & 정책**

**정의**: 제안 작성 시 지켜야 할 정부 지침 & 인건비 기준

**구조**:
```
A. 제안서 작성 가이드
  ├─ 포맷/형식 (페이지, 폰트)
  ├─ 표현 가이드 (금지어, 권장 표현)
  ├─ 품질 체크리스트
  └─ Shipley Color Team Review 프레임워크

B. 정부 정책 & 지침 (매우 중요)
  ├─ 예산편성지침 (기획재정부)
  ├─ 성과관리지침 (해당 부처)
  ├─ 예비타당성조사지침 (PIMAC)
  └─ 기술사업화평가 기준

C. 인건비 기준 (critical)
  ├─ 기획재정부 학술용역단가
  ├─ 컨설팅단가 (분야별)
  ├─ 급여 기준 (공무원, 대학 등)
  └─ 갱신일 & 적용 조건

D. 공공조달 규정
  ├─ 나라장터 제출 규칙
  ├─ 조달청 지침
  └─ 형식 요구사항

E. 평가기준 역설계 (참고)
  ├─ 공고유형별 평가배점 패턴
  └─ 발주처별 특성 (기술 vs 가격 민감도)
```

**사용 방식**:
```
제안서 작성 시작 → 해당 공고의 가이드라인 확인
  → 예산편성지침, 인건비 기준 적용
  → 제출 전 체크리스트 확인
```

**업데이트**:
- 반기별 + 정부 규정 변경 시 즉시

### ✅ **Category 7: 발주처 정보 & 관계 DB**

**정의**: 발주처와의 관계, 과거 경험, 담당자 정보 등

**구조**:
```
메타 DB:
  발주처명 | 담당부서 | 주력사업 | 연간공고수 | 과제규모평균 |
  우리관계(처음/경험있음/긴밀) | 성공경험(Y/N) | 실패경험(Y/N) |
  담당자명 | 담당자연락처 | 마지막거래일 | 비고

관계 이력:
  - 성공 수행한 과제 (링크: Category 2)
  - 실패한 과제 (링크: Category 10)
  - 예정된 과제 (관계 구축 기회)
```

**사용 방식**:
```
RFP 분석: "이 발주처를 우리가 잘 아는가?"
  ↓
KB에서 발주처 정보 확인:
  - "삼성과는 과거 3개 과제 성공"
  - "담당자: 김○○, 010-XXXX-XXXX"
  - "마지막 거래: 2025년 10월"
  
  ↓
제안 전략 수립:
  - 관계 기반 신뢰도 부각
  - 담당자 선제적 접촉 (기술 질문 등)
  - 과거 성공 사례 강조
```

**발주처별 특성 정보**:
```
국방부:
  - 관계 난이도: 높음 (초기 진입 어려움)
  - 보안 요구: 매우 높음
  - 평가기준: 기술 중시 (기술점수 편차 크다)
  - 우리 경험: 3건 성공, 2건 실패
  - 담당부서: OOO국 XXX과
  
삼성:
  - 관계 난이도: 중간
  - 가격 민감도: 높음
  - 평가기준: 가격 중시
  - 우리 경험: 5건 성공
  - 담당자: 이○○
  - 관계상태: 긴밀
```

**업데이트**:
- 거래 후 즉시 (관계 정보 갱신)
- 분기별 (담당자 변경 확인)

### ✅ **Category 8: 성과분석 & 입찰가 결정 데이터**

**정의**: 유사과제 낙찰가 분석 기반의 입찰가 결정 지원

**구조**:
```
A. 유사과제 낙찰가 DB (나라장터 자동 수집)
  프로젝트명 | 발주처 | 계약금액 | 낙찰가 | 낙찰률(%) | 
  포지셔닝 | 기술점수 | 가격점수 | 낙찰일

B. 입찰가 결정 분석
  ├─ 발주처별 낙찰가 범위 (평균, 최저, 최고)
  ├─ 포지셔닝별 승률 (offensive vs defensive)
  ├─ 금액대별 낙찰률 경향
  └─ "50억 범위 과제 평균 낙찰가: 46억 (92% 낙찰률)"

C. 입찰 의사결정 기록
  ├─ 예정 입찰가
  ├─ 실제 낙찰가
  ├─ 입찰가 적절도 (과다/적정/과소)
  └─ 사후분석 (의사결정 검증)
```

**사용 방식**:
```
입찰가 결정 단계
  ↓
유사과제 5-10건의 낙찰가 자동 추천
  ↓
분석 자료:
  - "50억 평균 낙찰가: 46억"
  - "Offensive 포지셔닝 낙찰률: 70%"
  - "국방부는 낙찰가 상대적으로 높음"
  
  ↓
입찰가 의사결정 지원
```

**업데이트**:
- 자동 (나라장터 크롤링)
- 실제 낙찰 후 (우리 입찰가 기록)

### 🆕 **Category 9: 유사과제 수행 기관 분석**

**정의**: "유사 분야의 어떤 기관이 과제를 수행했는가?" → 경쟁사/경쟁 전략 수립

**구조**:
```
메타 DB:
  과제명 | 발주처 | 분야 | 금액 | 낙찰기관명 | 수행기간 | 낙찰가 | 낙찰률

분석 뷰:
  의료 AI 분야 → 수행 기관 5곳
    ├─ 삼성SDS (2023, 2024, 2025 - 3건)
    ├─ 한글과컴퓨터 (2024 - 1건)
    ├─ LG (2023 - 1건)
    ├─ CJ (2022 - 1건)
    └─ ...

  기관별 특성 분석:
    ├─ 삼성: 주도적 수행 (의료AI 주력)
    ├─ 한글과: 가격 경쟁력 (낙찰가 낮은 편)
    └─ CJ: 최근 활동 감소
```

**사용 방식**:
```
RFP 분석: "의료 AI 과제가 들어왔다"
  ↓
KB에서 "의료AI 유사과제" 검색
  ↓
결과: 지난 3년간 수행 기관
  - 삼성SDS (주도, 기술력 높음)
  - 한글과컴퓨터 (가격 경쟁)
  - LG (한 번)
  
  ↓
경쟁 전략 수립:
  - "삼성과 기술로 경쟁할까? 가격으로 경쟁할까?"
  - "한글과와는 기술력으로 차별화하자"
  
  ↓
PT 전략:
  - 삼성의 강점 분석 (어떻게 이겼을까?)
  - 우리의 차별점 (기술 or 가격)
```

**업데이트**:
- 자동 (나라장터 크롤링과 동시)
- 분기별 (기관별 특성 분석 업데이트)

### 🆕 **Category 10: 리서치 & 기술 자료 (과제별 생명주기 관리)**

**정의**: 제안 작성 중 수집한 기술/시장/정책 자료 (수주 성공 시에만 보관)

**구조**:
```
A. 기술 자료
  ├─ 논문 (학술 자료)
  ├─ 특허 (기술 특허)
  ├─ 기술 가이드 (표준, 스펙)
  └─ 기술 사례 (구현 사례)

B. 시장/산업 분석
  ├─ 시장 보고서
  ├─ 경쟁 분석 자료
  ├─ 트렌드 분석
  └─ 고객 분석

C. 정책/규제 정보
  ├─ 정부 정책 문서
  ├─ 규제/가이드라인
  └─ 산업 표준

메타 DB:
  자료명 | 유형 | 관련과제 | 수주여부 | 수행시작 | 자료파일 | 
  활용도(높음/중간/낮음) | 기술노후도(신/중/구) | 비고
```

**생명주기 관리**:
```
[제안 단계]
  RFP 분석 → 기술/시장 리서치 → 자료 수집 (임시 저장)

[낙찰 결과에 따라 분기]

  Case A: 수주 성공 ✓
    → KB에 영구 보관 (프로젝트 수행 시 활용)
    → 태그: "수행중", "수행완료"
    → 메타: 활용도 기록 (높음/중간/낮음)
    
  Case B: 낙선 ✗
    → 삭제 (또는 6개월 후 자동 삭제)
    → 이유: 시의성 떨어짐, 재리서치 필요
```

**사용 방식**:
```
[제안 단계에서 사용]
제안 작성 → 기술 자료 검색 → 논문/보고서 참고

[프로젝트 수행 단계에서 사용]
수주 성공 → 기술 자료 활용 (구현 가이드로)

[다음 유사 제안 시]
"이 기술 자료 뭐였지?" → KB 검색 가능 (수주한 과제만)
```

**업데이트 정책**:
- **수주 성공 시**: 자동 저장 (그 때부터 영구 보관)
- **낙선 시**: 삭제 또는 6개월 후 자동 삭제
- **기술 노후도**: 분기별 확인 (기술 발전 추적)

**이점**:
- 낙선 자료로 인한 스토리지 낭비 방지
- 수주한 과제의 기술 자료는 프로젝트 수행에 즉시 활용
- 다음 유사 제안 시에도 참고 가능 (하지만 새로 리서치 필요)

## 4. 핵심 개선사항 (Before vs After)

### **Before (현재 산재 상태)**
```
실적증명서: 사업관리자 개인 PC 폴더
경력기술서: 매번 수작업 (서식 다양)
제출서류: 각 팀이 개별 관리
Q&A: 팀원 노트/메모 (조직화 안 됨)
회사정보: 마케팅/HR 분산 관리
가이드라인: 산발적 배포
발주처정보: 기록 없음
입찰가: 과거 데이터 활용 어려움
```

### **After (KB 중앙화)**
```
실적증명서: KB 파일 업로드 (중앙화, 검색 가능)
경력기술서: 경영기획실 배포 파일 + 서식 샘플 제공
제출서류: 공고유형별 템플릿 자동 추출
Q&A: 발표 대비 자동 검색 + 시사점 제시
회사정보: 분야별/고객별 차별성 자료 통합
가이드라인: 한 곳에 정책/인건비 기준 통합
발주처정보: 관계/담당자/경험 중앙 관리
입찰가: 유사과제 낙찰가 자동 분석 → 의사결정 지원
```
