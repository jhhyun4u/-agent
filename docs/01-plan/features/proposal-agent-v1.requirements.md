## 1. 제품 비전

### 1-1. TENOPA Intranet이란

TENOPA Intranet은 **용역 제안과 관련한 조직 지식을 축적·활용하는 사내 플랫폼**이다.
이 플랫폼 위에서 **AI Coworker**(에이전트)가 용역과제 모니터링과 제안서 공동 작성을 수행한다.

### 1-2. AI Coworker의 역할

AI Coworker는 도구(Tool)가 아닌 **동료(Coworker)**이다.

| 관점 | 도구(Tool) — 기존 프레이밍 | 동료(Coworker) — 현재 프레이밍 |
|------|--------------------------|-------------------------------|
| 관계 | 사람이 지시하면 AI가 실행 | 사람과 AI가 각자 역할로 협업 |
| 지식 | 매번 새로 시작 | **Knowledge Base에서 학습하고 축적** |
| 주도성 | 수동적 — 요청 시만 동작 | **능동적** — 과제 모니터링, 적합 공고 추천, 마감 알림 |
| 성장 | 변하지 않음 | **프로젝트를 거듭할수록 조직 지식과 함께 성장** |

**AI Coworker의 구체적 역할:**
1. **모니터링**: G2B 공고를 지속 감시하고 적합한 과제를 능동적으로 추천
2. **분석**: RFP를 구조화하고 KB의 과거 경험을 참조하여 전략 방향 제시
3. **공동 작성**: KB의 검증된 콘텐츠를 활용하여 제안서 초안을 작성, 사람이 리뷰·보완
4. **학습**: 매 프로젝트 완료 후 결과를 KB에 환류 — 수주 성공 패턴과 패찰 교훈 축적

### 1-3. 핵심 설계 철학

- **Knowledge-First**: 모든 제안 활동은 KB에서 시작하고, 결과는 KB로 돌아간다 — 조직 지식의 선순환
- **Coworker, Not Tool**: AI는 지시를 기다리지 않고, 과제를 모니터링하고 인사이트를 제안하는 능동적 동료
- **전략 우선**: 제안 작성 전 반드시 전략 수립 → 인간 승인 → 실행
- **포지셔닝 기반 전략**: 수성(Defensive) / 공격(Offensive) / 인접(Adjacent) 포지셔닝에 따라 전략 방향이 근본적으로 달라짐
- **Human-in-the-Loop**: 모든 단계 완료 후 반드시 Human 승인 게이트 통과
- **모듈 독립성**: 각 단계는 독립된 모듈로 설계, 개별 실행·재작업 가능
- **산출물 관리**: 모든 산출물은 버전 관리되어 언제든 참조·재작업 가능
- **병렬 처리**: AI Coworker가 독립적 작업을 동시 수행
- **전사 운영**: 팀장이 제안을 주도하고, 경영진이 파이프라인을 모니터링
- **성과 가시화**: 팀·부서·전사 수준의 제안 성과를 추적·분석
- **역할 기반 접근 제어**: 조직 내 역할에 따라 접근·승인·조회 권한을 차등 적용

### 1-4. 서비스 유형 및 사용 규모

- TENOPA 내부 직원 약 50명 (동시 접속이 아닌 순차 사용)
- 부서/팀 단위 운영, 싱글테넌트 (Organization 1개)
- SaaS 형태로 배포, MS365/Teams 연동

## 2. 조직 구조 및 역할

### 2-1. 조직 계층

```
회사 (Organization)
 └── 본부 (Division)          예: ICT사업본부, 공공사업본부
      └── 팀 (Team)           예: AI사업팀, 시스템사업팀
           └── 사용자 (User)   예: 홍길동 (팀원), 김팀장 (팀장)
```

### 2-2. 역할 정의

| 역할 | 코드 | 주요 권한 | 대상 범위 |
|------|------|-----------|-----------|
| **팀원** (Member) | `member` | 프로젝트 생성·작성, AI 실행, 피드백 반영, 본인 참여 프로젝트 조회 | 소속 팀 프로젝트 |
| **팀장** (Lead) | `lead` | 제안 프로세스 주도, 관심과제 선정, Go/No-Go 결정, 단계별 승인/반려, 팀원 배정, 팀 현황 조회 | 소속 팀 프로젝트 |
| **본부장** (Director) | `director` | Go/No-Go 최종 승인(금액 기준), 본부 전체 제안 현황 모니터링, 성과 대시보드 조회 | 소속 본부 전체 |
| **경영진** (Executive) | `executive` | 전사 모니터링, 전사 성과·수주율 분석, 전략적 의사결정 | 전사 |
| **관리자** (Admin) | `admin` | 사용자·조직 관리, 역량 DB 관리, 시스템 설정 | 시스템 전체 |

| 단계 | 팀원 (실행) | 팀장 (주도·승인) | 본부장 (승인·모니터링) | 경영진 (모니터링) |
|------|------------|-----------------|---------------------|------------------|
| STEP 0 관심과제 검색 | 검색 실행 | **관심과제 선정** | 모니터링 | 모니터링 |
| STEP 1① RFP 분석 | AI 실행·확인 | **분석 결과 승인** | 모니터링 | — |
| STEP 1② Go/No-Go | — | **Go/No-Go 결정** | **최종 승인** (3억↑) | **최종 승인** (5억↑) |
| STEP 2 제안전략 | 초안 작성 보조 | **전략 승인** | 모니터링 | — |

| STEP 4 정성제안서 | 섹션 작성 | **제안서 승인** | **최종 승인** (3억↑) | — |
| STEP 5 PPT | 슬라이드 작성 | **최종 승인** | 모니터링 | — |

### 2-4. 결재선 규칙

| 기준 | 결재선 |
|------|--------|
| 예산 3억 미만 | 팀장 단독 승인 |
| 예산 3억 이상 ~ 5억 미만 | 팀장 → **본부장** 승인 필수 |
| 예산 5억 이상 | 팀장 → 본부장 → **경영진** 승인 필수 |

> 결재선은 STEP 1② Go/No-Go와 STEP 4 제안서 최종 승인에 적용된다.
> 다른 단계의 승인은 팀장 단독 권한이다.

**자기결재 방지 원칙:**
- 프로젝트 생성자(제안 담당자)가 팀장인 경우, **본인이 생성한 프로젝트의 승인을 본인이 수행할 수 없다**
- 이 경우 결재선은 자동으로 **차상위 결재자(본부장)**로 상향된다
- 예산 기준과 무관하게 자기결재 방지 원칙이 우선 적용된다

### 2-5. 인증

| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| AUTH-01 | Azure AD(Entra ID) SSO를 통한 MS365 계정 로그인 | Must |
| AUTH-02 | Supabase Auth가 Azure AD를 OAuth provider로 연동 | Must |
| AUTH-03 | 로그인 시 조직·역할 자동 매핑 (Azure AD 그룹 → 시스템 역할) | Must |
| AUTH-04 | 세션 관리 (JWT, 자동 갱신, 타임아웃 30분) | Must |
| AUTH-05 | 비로그인 접근 차단 | Must |
| AUTH-06 | **세션 만료와 AI 작업 분리** — 세션이 만료되더라도 서버 측에서 실행 중인 AI 작업은 중단 없이 백그라운드 계속 수행. 재로그인 시 진행 중/완료된 AI 작업 결과를 자동 복원하여 표시 | Must |

### 2-6. 데이터 접근 제어

| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| ACL-01 | 팀원은 본인이 참여하는 프로젝트만 조회·편집 | Must |
| ACL-02 | 팀장은 소속 팀의 모든 프로젝트 조회·편집·승인 | Must |
| ACL-03 | 본부장은 소속 본부의 모든 프로젝트 조회·승인 (편집은 불가) | Must |
| ACL-04 | 경영진은 전사 프로젝트 조회 (편집·승인은 결재선 해당 시만) | Must |
| ACL-05 | 관리자는 모든 프로젝트·사용자·설정에 접근 | Must |
| ACL-06 | PostgreSQL RLS(Row Level Security)로 DB 수준 격리 | Must |

### 2-7. 사용자 라이프사이클 관리

> **원칙**: 권한을 가진 사용자(admin)가 사용자 정보를 관리한다.

| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| ULM-01 | 사용자 등록 — admin이 이름·이메일·소속(본부/팀)·역할을 지정하여 생성 | Must |
| ULM-02 | Azure AD 계정과 시스템 사용자 자동 연결 (AUTH-03 그룹 매핑 기반) | Must |
| ULM-03 | 소속 변경 (팀 이동) — admin이 사용자의 팀/본부 변경, 기존 프로젝트 접근 유지 여부 선택 | Must |
| ULM-04 | 역할 변경 — admin이 사용자 역할 승격/변경 (예: member → lead) | Must |
| ULM-05 | 팀장 교체 — 기존 팀장 → member 전환 + 신규 팀장 지정, 진행 중 결재 자동 재배정 | Must |
| ULM-06 | 퇴사 처리 — 계정 비활성화(soft delete), 진행 중 프로젝트 담당 재배정 알림 | Must |
| ULM-07 | 임시 위임 — 팀장 부재 시 특정 기간 승인 권한을 지정 팀원에게 위임 | Should |
| ULM-08 | 계정 비활성화/재활성화 — 휴직·장기 부재 시 계정 잠금 및 복원 | Should |
| ULM-09 | 사용자 목록 조회·검색 — admin 전용, 소속/역할/상태 필터 | Must |

#### 퇴사·이동 시 처리 흐름

```
퇴사 처리:
1. admin이 사용자 비활성화 실행
2. 해당 사용자가 담당 중인 프로젝트 목록 표시
3. 프로젝트별 후임자 지정 (또는 미배정 상태로 전환)
4. 진행 중 결재선에 포함된 경우 → 후임자에게 결재 재배정
5. Azure AD 계정 비활성화와 연동 (수동 또는 자동 감지)

팀 이동 처리:
1. admin이 소속 팀/본부 변경
2. 기존 프로젝트 접근: 유지(읽기전용) 또는 해제 선택
3. 새 팀의 프로젝트에는 팀장이 별도 참여자 배정
```

## 3. 입찰 포지셔닝 전략 개념

> **모든 제안 전략의 출발점**: 우리가 이 입찰에서 어느 포지션인지에 따라 Ghost Theme, Win Theme, 가격 전략이 완전히 달라진다.

### 3-1. 포지셔닝 유형 정의

| 유형 | 정의 | 전형적 상황 |
|------|------|------------|
| **수성형 (Defensive)** | 우리가 잘하고 있는 영역. 현재 수행 중이거나 직접 연관된 후속 사업 | 기존 고객 재입찰, 직접 연속 사업 |
| **공격형 (Offensive)** | 신규 진입 영역. 처음 도전하는 발주기관 또는 새로운 기술·서비스 영역 | 첫 공략 고객, 신기술 영역 |
| **인접형 (Adjacent)** | 관련 있지만 직접 수행한 적 없는 영역. 유사 실적은 있으나 동일 분야는 아님 | 타 부처 유사 사업 경험 → 신규 부처 공략 |

### 3-2. 포지셔닝별 전략 매트릭스

| 전략 요소 | 수성형 (Defensive) | 공격형 (Offensive) | 인접형 (Adjacent) |
|----------|-------------------|-------------------|------------------|
| **Ghost Theme 방향** | "교체의 리스크 — 왜 바꾸면 위험한가" | "현재 담당사·방식의 한계 — 왜 바꿔야 하는가" | "타 기관 성공 사례 — 이제 우리 기관도 적용할 때" |
| **Win Theme 핵심** | 검증된 수행 실적 + 연속성 + 인수인계 비용 제거 | 혁신적 접근법 + 기술 차별화 + 비용 효율성 | 유사 실적 적극 부각 + 빠른 적용 + 전환 비용 최소화 |
| **Action Forcing Event** | "재입찰 지연 시 서비스 공백 발생" | "경쟁사 선점 전 기술 격차 확대" | "타 기관 성공 후 확산 흐름에 합류할 적기" |
| **가격 전략** | 현행 유지 수준 (프리미엄 가능) | 진입 전략 가격 (낮게) 또는 혁신 가치 premium | 중간 수준 (리스크 감수 가격) |
| **인력 구성 강조** | 現 수행 인력 연속 투입 + 축적 노하우 | 핵심 기술 전문가 전진 배치 | 유사 사업 경험자 + 현장 적응력 |
| **실적 활용** | 동일 사업 수행 실적 전면 부각 | 최신 기술 적용 사례 + 혁신 레퍼런스 | 유사 분야 실적 → 전이 가능성 논리화 |
| **리스크 메시지** | "교체 리스크 없는 안전한 선택" | "변화의 리스크보다 현상 유지의 리스크가 크다" | "검증된 방법론으로 적용 리스크 최소화" |

### 3-3. Win Theme 방정식 (포지셔닝 적용)

┌───────────────────────────────────────────────────────────────┐

| CLG-03 | 게시된 콘텐츠 수정 시 신규 버전 생성 + 팀장 재승인 | Should |
| CLG-04 | 콘텐츠 폐기 — 더 이상 사용하지 않는 콘텐츠 아카이브 처리 (팀장/admin) | Should |

#### 4-B-4. 콘텐츠 라이브러리 AI 활용 방식

```
STEP 4 정성제안서 작성 시:
1. AI가 섹션 주제 분석 → 콘텐츠 라이브러리에서 유사 콘텐츠 시맨틱 검색
2. 품질 점수 높은 순으로 후보 콘텐츠 제시
3. 사용자가 참조할 콘텐츠 선택 (또는 AI가 자동 참조)
4. AI가 선택된 콘텐츠를 기반으로 현재 RFP 맥락에 맞게 재작성
5. 출처 콘텐츠 ID를 산출물에 기록 → 수주 시 기여도 추적에 활용

STEP 2 제안전략 시:
- 유사 프로젝트의 이전 전략 서술을 참조하여 전략 초안 품질 향상

제안 완료 후:
- 수주 확정 시 → 해당 제안서 콘텐츠 자동 분석 → 라이브러리 등록 후보 추천
- 패찰 시 → 해당 콘텐츠 품질 점수에 반영 (가중치 감소)
```

#### AI 자가진단 평가 항목
```
1. Compliance Matrix 체크 (자동)

2. 전략 반영도 체크
   - Win Theme이 각 섹션에 반영되었는가
   - Hot Button이 적절히 부각되었는가
   - Action Forcing Event가 표현되었는가
   - 스토리라인 4단계 구조 준수 여부

3. 품질 체크
   - 수치·근거 구체성 (추상 표현 감지)
   - 발주처 언어 사용 여부
   - 섹션 간 논리 일관성
   - 경쟁 차별성 명확도

4. ★ v4.9 근거 신뢰성 체크 (TRS-07 연동)
   - 출처 없는 사실 주장 탐지 (수행실적·인력·금액 등)
   - 출처 태그 누락 수치 목록화 ([KB]/[RFP]/[G2B]/[추정] 미부착 수치)
   - 과장 표현 감지 (TRS-10: "업계 최초", "혁신적" 등 → 구체 수치 대체 권고)
   - 비RFP 용어 사용 감지 (TRS-11)
   - 섹션 간 수치 일관성 체크 (TRS-08)
   - KB 참조 비율 산출 (KB 기반 문장 수 / 전체 문장 수)

5. 종합 점수 및 판정
   - 100점 만점 환산 점수 (4축 가중 합산)
   - 배점: Compliance 25점 + 전략 반영도 25점 + 품질 25점 + 근거 신뢰성 25점
   - 통과 기준: 80점 이상 (미달 시 개선 항목 강조)
   - 근거 신뢰성 15점 미만 시 개별 경고 ("출처 보강 필요")
```

| STEP 5 PPT | iterate 완료 후 최종 승인 | 작업 완료 확정 |

### 8-1. 프로젝트 상태 머신

> **프로젝트의 전체 생명주기 상태를 정의한다. 워크플로 STEP과는 별도로 프로젝트 수준의 상태를 관리.**

```
                        ┌─────────┐
                        │  draft  │  ← 프로젝트 생성 직후
                        └────┬────┘
                             │ STEP 0 실행
                             ▼
                        ┌──────────┐
                        │ searching│  ← 공고 검색 중
                        └────┬─────┘
                             │ 공고 Pick-up
                             ▼
                        ┌───────────┐
                        │ analyzing │  ← RFP 분석 중
                        └────┬──────┘
                             │ Go 결정
                             ▼
                     ┌───────────────┐
          No-Go ←────│  strategizing │  ← 전략·계획·작성 중 (STEP 2~5)
          │          └───────┬───────┘
          ▼                  │ STEP 5 최종 승인 → 제출
     ┌─────────┐             ▼
     │ no_go   │        ┌───────────┐
     └────┬────┘        │ submitted │  ← 제안서 제출 완료
          │             └────┬──────┘
          │                  │ 발표 완료
          │                  ▼
          │             ┌─────────────┐
          │             │ presented   │  ← 발표 완료 + Q&A 기록 입력
          │             └────┬────────┘
          │                  │ 결과 확정 (선정/탈락)
          │             ┌────┴────┐
          │             ▼         ▼
          │        ┌─────┐   ┌──────┐
          │        │ won │   │ lost │
          │        └──┬──┘   └──┬───┘
          │           │         │
          │           └────┬────┘
          │                ▼
          │         ┌────────────┐
          └────────►│ retrospect │  ← 회고 완료 후 KB 환류
                    └────────────┘

    예외 상태:
    ┌──────────┐  ← 공고 취소/내부 사정으로 일시 보류
    │ on_hold  │  → 어느 상태에서든 전환 가능, 이전 상태로 복원 가능
    └──────────┘

    ┌──────────┐  ← 제출 기한 초과 시 자동 전환
    │ expired  │  → searching, analyzing, strategizing에서 기한 초과 시
    └──────────┘     → 회고(retrospect) 대상 — 왜 기한 내 제출하지 못했는지 교훈 축적

    ┌───────────┐  ← 중도 중단 (사용자가 명시적으로 작업 포기)
    │ abandoned │  → draft, searching, analyzing, strategizing에서 전환 가능
    └───────────┘

    ※ no_go, abandoned, expired 모두 회고(retrospect) 대상 — 의사결정·중단·기한초과 사유가 교훈으로 축적됨
```

| ID | 요구사항 | 우선순위 |
|----|----------|----------|

### 12-0. AI 산출물 신뢰성 정책

> **원칙**: 용역 제안서는 발주처 평가위원이 검증하는 공식 문서다. AI Coworker가 생성하는 모든 산출물은 **사실에 기반(Grounded)**하고, **출처를 명시**하며, **수치 데이터를 검증 가능한 형태**로 제시해야 한다. 할루시네이션(사실이 아닌 내용 생성)은 제안서 탈락의 직접적 원인이 되므로 시스템적으로 방지한다.

#### 12-0-1. 할루시네이션 방지 규칙

| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| TRS-01 | **KB 우선 참조 원칙** — AI가 제안서 텍스트를 생성할 때, KB(역량 DB·콘텐츠 라이브러리·발주기관 DB·경쟁사 DB)에 관련 데이터가 존재하면 **반드시 KB 데이터를 우선 인용**. KB에 없는 내용을 AI가 자체 생성(일반 지식 기반)할 경우 해당 부분에 `[일반지식]` 태그를 부착 | Must |
| TRS-02 | **사실 주장 금지 규칙** — AI는 자사 수행실적·인력·기술 관련 구체적 사실(프로젝트명, 인원수, 금액, 기간 등)을 KB에 없는 상태에서 **임의로 생성하지 않음**. KB에 해당 데이터가 없으면 `[KB 데이터 필요: 수행실적 정보를 입력해 주세요]` 형태의 플레이스홀더를 삽입 | Must |
| TRS-03 | **수치 데이터 출처 필수** — 산출물에 포함되는 모든 수치(금액, 비율, 건수, 기간 등)는 **출처 태그** 부착 필수. 출처 유형: `[KB]` (KB 데이터 기반), `[RFP]` (RFP 원문 인용), `[G2B]` (나라장터 데이터), `[추정]` (AI 추론, 근거 명시) | Must |
| TRS-04 | **외부 데이터 인용 규칙** — AI가 시장 동향·기술 트렌드·통계 등 외부 정보를 인용할 경우, **공신력 있는 출처**(정부 통계, 학술 논문, 공공기관 보고서, 업계 공식 보고서 등)만 허용. 개인 블로그·비공식 매체·출처 불명 데이터 인용 금지. 출처를 특정할 수 없는 일반 상식은 `[일반지식]` 태그 | Must |
| TRS-05 | **KB 참조 신뢰도 3단계 표시** — 모든 AI 산출물에 참조 신뢰도를 표시: ① `KB 기반` (KB 데이터로 충분히 뒷받침), ② `혼합` (일부 KB + 일부 일반지식), ③ `일반지식 기반` (KB 참조 없음). 기존 Cold Start 대응 원칙(§4)과 연동 | Must |

#### 12-0-2. 출처 추적 및 검증 지원

| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| TRS-06 | **인라인 출처 표기** — AI가 생성한 제안서 텍스트 내에서 주요 주장·수치마다 인라인 출처 마커 삽입 (예: `[역량DB-PRJ-023]`, `[RFP p.12]`, `[G2B 낙찰정보]`). Human 리뷰어가 클릭하면 원본 데이터로 이동 가능 | Must |

| VAL-03 | 텍스트 입력 (프로젝트명 등 단일 필드) | 최대 200자, 특수문자 제한 (HTML 태그 불허) | Must |
| VAL-04 | 텍스트 입력 (제안서 본문, 피드백 등 장문) | 최대 10,000자, HTML 태그 이스케이프 | Must |
| VAL-05 | 필수 입력 필드 | 프로젝트 생성 시: 프로젝트명, RFP 파일. Go/No-Go 시: 결정사유. 승인/반려 시: 사유 | Must |
| VAL-06 | 숫자 입력 (예산, 인력 수 등) | 양수만 허용, 예산은 0원 초과 ~ 999억 이하 | Must |
| VAL-07 | CSV/Excel 일괄 업로드 | 필수 컬럼 존재 확인, 행 수 상한 1,000행, 데이터 타입 검증 | Should |

## 14. 용어 정의

| 용어 | 정의 |
|------|------|
| TENOPA Intranet | 용역 제안과 관련한 조직 지식을 축적·활용하는 사내 플랫폼 |
| AI Coworker | TENOPA Intranet 위에서 과제 모니터링과 제안서 공동 작성을 수행하는 AI 에이전트 — 도구가 아닌 동료 |
| Knowledge Base (KB) | 역량 DB·콘텐츠 라이브러리·발주기관 DB·경쟁사 DB·교훈 아카이브를 포함하는 조직 지식의 총체 |
| 학습 피드백 루프 | 프로젝트 완료 후 결과(수주/패찰)를 KB에 환류하여 다음 제안 품질을 높이는 순환 구조 |
| 회고 (Post-Mortem) | 프로젝트 완료 후 전략 효과·부족한 점·개선 포인트를 구조화하여 기록하는 과정 |
| RFP | 제안요청서 (Request for Proposal) |
| 산출물 (Artifact) | 각 단계에서 생성되는 저장 가능한 결과물 |
| Win Theme | "왜 우리인가?"를 답하는 핵심 차별화 메시지 (자사역량 × 발주처니즈 × 경쟁사약점) |
| Ghost Theme | 발주처가 직면한 근본 문제 (표면적 요구 이면의 진짜 고민) |
| Action Forcing Event | 지금 이 제안사를 선택해야 하는 시급한 이유 |
| Hot Button | 발주처가 가장 민감하게 여기는 핵심 관심사 |
| Human 게이트 | 사람이 확인·승인해야 다음 단계로 진행되는 체크포인트 |
| Compliance Matrix | RFP 요구사항 항목과 제안서 대응 여부를 매핑한 체크리스트 |
| iterate | 피드백을 반영하여 반복적으로 개선하는 과정 |
| G2B | 나라장터 국가종합전자조달시스템 |
| 케이스 A | RFP에 작성 서식이 없는 경우 (자유 양식) |
| 케이스 B | RFP에 작성 서식이 있는 경우 (서식 준수) |
| SSE | Server-Sent Events — 서버에서 클라이언트로의 단방향 실시간 스트리밍 |
| RLS | Row Level Security — PostgreSQL의 행 단위 접근 제어 정책 |
| 결재선 | 금액 기준에 따른 승인 권한 경로 (팀장 → 본부장 → 경영진) |
| 수주율 | Go 결정 건수 대비 실제 낙찰 건수의 비율 |
| Azure AD (Entra ID) | Microsoft의 기업용 ID/인증 서비스 — MS365 SSO 기반 |
| 수성형 (Defensive) | 우리가 잘 알고 잘 하는 영역 — 현재 수행 중이거나 직접 연관된 후속 사업 |
| 공격형 (Offensive) | 신규 진입 영역 — 처음 도전하는 발주기관 또는 새로운 기술·서비스 영역 |
| 인접형 (Adjacent) | 관련 있지만 직접 수행한 적 없는 영역 — 유사 실적으로 전이 논리 적용 |
| 입찰 포지셔닝 | 우리 회사의 해당 입찰 내 경쟁 위치 — 전략 방향을 결정하는 최우선 분류 변수 |
| 프로젝트 상태 머신 | 프로젝트의 생명주기 상태 흐름 (draft→searching→analyzing→strategizing→submitted→presented→won/lost→retrospect) |
| presented | 제안서 제출 후 발표를 완료한 상태 — Q&A 기록 입력 대상 |
| abandoned | 중도 중단 — 제안 작업 중 명시적으로 포기한 프로젝트 |
| 발주기관 DB | 발주기관별 관계 수준·평가 성향·과거 입찰 이력을 축적하는 지식 영역 |
| 경쟁사 DB | 경쟁사별 강점·약점·가격 패턴·대응 전략 이력을 축적하는 지식 영역 |
| 통합 검색 | KB의 모든 영역(역량·콘텐츠·발주기관·경쟁사·교훈)을 시맨틱+키워드로 한 번에 검색하는 기능 |
| 사용자 라이프사이클 | 사용자 등록부터 퇴사·비활성화까지의 계정 관리 흐름 |
| 온보딩 | 시스템 최초 배포 시 조직·사용자·역량DB 등 초기 데이터를 설정하는 과정 |
| 콘텐츠 라이브러리 | 이전 제안서에서 검증된 재사용 가능한 텍스트 블록·답변·섹션의 저장소 |
| 시맨틱 검색 | 키워드 일치가 아닌 의미 유사도 기반 검색 (임베딩 벡터 활용) |
| Prompt Caching | Anthropic이 제공하는 프롬프트 캐싱 기능 — 동일 프롬프트 접두사 재사용 시 캐시된 토큰의 input 비용 90% 절감 |
| Cold Start | KB 데이터가 부족한 시스템 초기 상태 — AI 추천·전략 품질이 제한적이며, 데이터 축적에 따라 점진적 개선 |
| 크로스팀 프로젝트 | 주관 팀(1개) + 참여 팀(복수)이 함께 수행하는 제안 프로젝트. 주관 팀장이 총괄, 참여 팀원은 섹션 담당자로 배정 |
| 콘텐츠 품질 점수 | 수주 기여도 + 재사용 횟수 + 최신성을 종합한 콘텐츠 가치 지표 |
| soft delete | 데이터를 실제 삭제하지 않고 비활성화 플래그로 표시하는 방식 |
| graceful degradation | 외부 서비스 장애 시 전체 서비스를 중단하지 않고 해당 기능만 비활성화하는 방식 |
| 할루시네이션 (Hallucination) | AI가 사실이 아닌 내용을 실제인 것처럼 생성하는 현상 — 제안서에서는 허위 수행실적·가공 수치·존재하지 않는 인력 정보 등이 해당 |
| Grounding (근거 기반 생성) | AI 출력을 KB·RFP·G2B 등 검증된 데이터에 기반하여 생성하는 방식 — 할루시네이션 방지의 핵심 기법 |
| 출처 태그 | AI 산출물의 주장·수치에 부착하는 근거 표시 — `[KB]`, `[RFP]`, `[G2B]`, `[추정]`, `[일반지식]` 유형 |
| 인라인 출처 마커 | 제안서 텍스트 내에서 원본 데이터를 참조할 수 있는 링크형 표기 (예: `[역량DB-PRJ-023]`) |
