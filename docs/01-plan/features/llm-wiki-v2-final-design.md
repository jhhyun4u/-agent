# Knowledge Base v2.0 — 최종 설계 (Proposal Workflow 통합)

**Date:** 2026-04-14  
**Version:** Final Design (Corrected)  
**Based on:** 실제 제안 워크플로우 & 운영 프로세스

---

## 1. 제안 워크플로우 내 KB 역할

```
[제안 워크플로우]

Step 0: RFP 발굴
  ↓ (KB 검색: 유사과제 자동 추천)
Step 1: RFP 분석 & Go/No-Go
  ├─ RFP 분석 (자동)
  ├─ 유사과제 낙찰가 분석 (Category 7)
  ├─ 경쟁사 분석 (Category X: 경쟁사 정보)
  ├─ 발주처 특성 분석 (Category X: 발주처 정보)
  └─ Go/No-Go 판정
  ↓
Step 2: 제안 전략 수립
  ├─ 포지셔닝 결정 (Category 7: 포지셔닝별 승률)
  ├─ 입찰가 결정 (Category 7: 유사과제 낙찰가 분석)
  └─ Win Theme 정의
  ↓
Step 3: 제안서 작성
  ├─ 인력 선정 (Category 1: 인력DB)
  ├─ 유사과제 실적 수집 (Category 2: 실적목록 + 실적증명서/계약서)
  ├─ 회사 강점 부각 (Category 5: 차별성 자료)
  ├─ 제출서류 준비 (Category 3: 템플릿 + 경력기술서)
  ├─ 기술 참고 (Category X: 리서치/기술 자료)
  ├─ 지침 확인 (Category 6: 가이드라인)
  └─ 제안서 초안 완성
  ↓
Step 4: 내부 리뷰 & 품질 검증
  ├─ 체크리스트 (Category 6: 가이드라인)
  └─ 가이드라인 준수 확인
  ↓
Step 5: 발표 준비
  ├─ 예상 Q&A 준비 (Category 4: Q&A 아카이브)
  └─ 시사점 확인 (Category 4: 발표 시사점)
  ↓
Step 6: 결과 & 학습
  ├─ Q&A 기록 (→ Category 4)
  ├─ 낙찰가 기록 (→ Category 7)
  ├─ 평가점수 기록 (→ Category 7)
  └─ 사후분석 (→ 가이드라인 개선)
```

## 2. 최종 Knowledge Base 구조 (10개 Category)

### ✅ **Category 1: 인력 자산 (People Asset Database)**

**포함 내용:**
- 직원 정보: 이름, 직급, 전공, 자격증, 경력연수
- 주요 경력 프로젝트
- 담당 역할, 전문분야
- 언어능력

**사용 방식:**
- RFP 요구인력 조건 매칭
- 경력기술서 기초 정보 추출 (프로필)

**주의:**
- 경력기술서 **자동생성 아님**
- 각 공고별 서식에 맞게 담당자가 수작업 작성
- KB는 인력 정보만 제공 (프로필 역할)

**업데이트:**
- 분기별 (HR + 개인 검증)

**메타:**
```
인력명 | 직급 | 전공 | 자격증 | 경력연수 | 주요경력 | 담당역할 | 언어능력 | 최신화일
```

---

### ✅ **Category 2: 수행실적 자산 (Project Portfolio with Proof of Performance)**

#### 2-1. 수행실적 목록 (추천 & 필터링용)
```
프로젝트명 | 발주처 | 계약금액 | 기간 | 담당팀 | 주요성과 | 수행일자
```

**사용 방식:**
- RFP 분석 → 유사과제 자동 추천
- 금액, 분야, 발주처 등으로 필터링
- "이 프로젝트를 실적으로 활용하자" → 담당자가 실적증명서/계약서 찾음

#### 2-2. 실적증명서 & 계약서 보관 (원본 관리)
**중요**: 발주처가 발급한 실적증명서 또는 계약서 원본/사본 보관

```
프로젝트명 | 발주처 | 실적증명서 (링크) | 계약서 (링크) | 보관위치 | 관리담당자
```

**관리 방식:**
- 각 과제별로 실적증명서 또는 계약서 원본 보관
- 제안 시 필요하면 해당 파일 꺼내서 활용
- **재사용** (자동생성 아님)

**업데이트:**
- 프로젝트 완료 후 즉시 (PM)
- 분기말 검증 (PM)

---

### ✅ **Category 3: 제출서류 템플릿 (Submission Document Templates)**

#### 3-1. 포함 내용
```
A. 법률/행정 서식
  ├─ 비밀유지서약서 (보안등급별: 일반/상대비밀/국방)
  ├─ 참여현황서
  ├─ 자격요건 확인서
  ├─ 기술안전관리계획
  └─ 기타 정부 규정 서식

B. 공고유형별 서식
  ├─ 정부과제 (나라장터 표준)
  ├─ 국방과제 (보안 관련)
  ├─ 연구용역 (과제관리비, 성과관리)
  └─ 민간과제 (자유 포맷)

C. 경력기술서 서식 샘플
  ├─ 정부과제 표준 양식
  ├─ 국방과제 양식
  └─ 민간과제 자유 포맷
```

#### 3-2. 사용 방식
```
RFP 업로드
  ↓ (자동 분류)
공고유형 식별 (정부/국방/민간)
  ↓
필요 서식 자동 추출
  ↓
템플릿 + 가이드 제시
```

**업데이트:**
- 반기별 (정부 규정 변경)
- 즉시 (새로운 규정/공고유형)

---

### ✅ **Category 4: Q&A 아카이브 & 발표 시사점**

#### 4-1. 구조
```
분류 | 발주처 | 프로젝트명 | 질문 | 우리답변 | 평가위원반응 | 낙찰여부 | 기술점수 | 작성일
```

#### 4-2. 발표 시 활용
```
발표 예정 "국방부 AI 이상탐지"
  ↓
KB 검색 (발주처 + 분야)
  ↓
결과 반환:
1. 유사 과제 Q&A 5건
2. 시사점: "보안 질문 자주 나옴 → 기술점수 영향 큼"
3. 성공 답변 템플릿
```

**업데이트:**
- 발표 직후 (질문 기록)
- 결과 공시 후 (평가점수 함께 기록)

---

### ✅ **Category 5: 회사정보 & 차별성 자료 (Company Profile & Differentiation Assets)**

#### 5-1. 기본 회사정보
```
회사 소개 (역사, 비전, 규모)
조직도
핵심 역량 (주력 분야)
사업 실적 (매출, 성장률)
인증/자격 (ISO, 특허, 수상)
```

#### 5-2. 차별성 부각 자료 (신규 & 중요)
**제안마다 다르게 강조할 수 있는 기초자료들**

```
A. 핵심역량 분야별 사례
  ├─ IT/클라우드 분야 강점 사례
  ├─ 의료/헬스케어 분야 강점 사례
  ├─ 국방/보안 분야 강점 사례
  └─ ... (분야별)

B. 차별화 포인트 DB
  ├─ 기술력 (기술 혁신, R&D 투자 사례)
  ├─ 인력 (전문 인력, 교육 투자)
  ├─ 경험 (유사과제 성공 사례)
  ├─ 가치 (가성비, 고객만족도)
  └─ 신뢰 (신용등급, 계약이행률)

C. 고객별 강점 맵
  ├─ 삼성과제 → "삼성과의 협력 경험, 대규모 프로젝트 경험" 강조
  ├─ 국방부과제 → "보안, 신뢰성" 강조
  ├─ 정부과제 → "컴플라이언스, 공공성" 강조
  └─ ...

D. 시각화 자료
  ├─ 회사 성장 그래프
  ├─ 분야별 매출 비중
  ├─ 수상/인증 타임라인
  ├─ 주요 프로젝트 포트폴리오
  └─ 팀별 전문성 인포그래픽
```

#### 5-3. 사용 방식
```
제안서 작성 중: "우리 회사의 강점을 뭘로 어필할까?"
  ↓
RFP 분석 + 경쟁사 분석
  ↓
"이 발주처는 기술력 중시" → 기술력 사례 자료 선택
"이 과제는 신뢰성이 중요" → 신뢰 관련 자료 선택
  ↓
차별성 자료 활용 → 제안서에 반영
```

**업데이트:**
- 월별 (회사 정보 변경 시)
- 분기별 (자료 업데이트)

---

### ✅ **Category 6: 가이드라인 & 정책 (Guidelines & Policies)**

#### 6-1. 제안서 작성 가이드
```
제안서 포맷/형식 (페이지, 폰트, 여백)
제안서 표현 가이드 (금지어, 권장 표현)
품질 체크리스트 (제출 전 확인사항)
Shipley Color Team Review 프레임워크
```

#### 6-2. 정부 정책 & 지침 (매우 중요)
```
A. 발주처 별 요구 지침
  ├─ 예산편성지침 (기획재정부)
  ├─ 성과관리지침 (해당 부처)
  ├─ 예비타당성조사지침 (PIMAC)
  └─ 기술사업화평가 기준 (R&D)

B. 인건비 기준
  ├─ 기획재정부 학술용역단가
  ├─ 컨설팅단가 (분야별)
  ├─ 급여 기준 (공무원, 대학 등)
  └─ 갱신일 및 적용 조건

C. 공공조달 규정
  ├─ 나라장터 제출 규칙
  ├─ 조달청 지침
  ├─ 형식 요구사항 (분량, 서명, 기밀마크)
  └─ 제출 금지사항
```

#### 6-3. 평가기준 역설계 (추가)
```
공고유형별 평가배점 패턴
  ├─ 정부R&D: 기술 60% vs 가격 40% (평균)
  ├─ 국방: 기술 70% vs 가격 30%
  └─ 민간: 기술 40% vs 가격 60%

발주처별 특성
  ├─ "국방부는 기술점수 편차 큼 (50-95점)"
  ├─ "삼성과제는 가격 민감"
  └─ "정부과제는 규정 준수 매우 엄격"
```

**업데이트:**
- 반기별 (정부 정책 개정)
- 즉시 (새로운 지침 발표)

## 3. 추가 필요 Category (신규 제안)

사용자의 요청: "여기에 한정하지 않아서 필요한 부분이 있다면 제안해줘"

제안 워크플로우를 완벽히 지원하려면, 다음이 추가로 필요해 보입니다:

---

### 🆕 **Category 7: 리서치 & 기술 자료 (Research & Technical Resources)**

#### 7-1. 필요성
RFP 분석 & 제안 작성 단계에서:
- 기술 타당성 검증 (논문, 학술 자료)
- 시장/산업 분석 (시장 동향, 경쟁사 현황)
- 규제/정책 분석 (정부 정책, 법규)

#### 7-2. 포함 내용
```
A. 기술 자료
  ├─ 논문 (학술 논문, 백서)
  ├─ 특허 (관련 기술 특허)
  ├─ 기술 스펙 (표준, 가이드)
  └─ 사례 연구 (기술 적용 사례)

B. 시장/산업 분석
  ├─ 시장 규모 & 동향
  ├─ 경쟁사 현황 (분석 보고서)
  ├─ 고객 니즈 분석
  └─ 기술 로드맵

C. 규제/정책 정보
  ├─ 관련 법규 (데이터보호, 개인정보보호, 보안)
  ├─ 정부 정책 (디지털전환, AI, 클라우드 등)
  └─ 업계 가이드라인
```

#### 7-3. 사용 방식
```
RFP 분석: "이 기술이 타당한가?"
  ↓
KB 리서치 자료 검색
  ↓
결과: 관련 논문 3건, 사례 2건 제시
  ↓
기술 타당성 검증 + 제안서 작성에 반영
```

#### 7-4. 유지관리
**우선순위**: **중간** (재사용 빈도 낮음, 하지만 필요할 때 필수)
- 비용 대비 효과: 저 (검색에만 활용, 재사용성 낮음)
- 수집 난이도: 중 (외부 자료 수집 필요)

**제안**: Phase 2 또는 Phase 3에서 수집 (초기 Phase 1에서는 생략 가능)

---

### 🆕 **Category 8: 경쟁사 & 시장 정보 (Competitive Intelligence)**

#### 8-1. 필요성
전략 수립 & 제안 차별화 단계에서:
- 경쟁사가 누구인가?
- 경쟁사의 강점/약점은?
- 우리의 차별화 포인트는?

#### 8-2. 포함 내용
```
A. 경쟁사 DB
  ├─ 주요 경쟁사 (국내 top 5-10)
  ├─ 경쟁사 프로필 (사업분야, 규모, 강점)
  ├─ 경쟁사 수주 현황 (최근 낙찰 사례)
  └─ 경쟁사 기술력 평가

B. 경쟁 분석 보고서
  ├─ 분야별 경쟁 현황
  ├─ 경쟁사 포지셔닝 맵
  ├─ 우리 vs 경쟁사 강점/약점
  └─ 차별화 전략
```

#### 8-3. 사용 방식
```
RFP 분석: "경쟁사가 누구일까?"
  ↓
KB에서 유사과제의 낙찰사 확인
  ↓
경쟁사 프로필 & 강점 분석
  ↓
"이 경쟁사는 가격 경쟁력이 강하다 → 기술로 차별화하자"
```

#### 8-4. 유지관리
**우선순위**: **중간** (전략 수립에 필요)
- 비용 대비 효과: 중 (전략 수립에 직접 영향)
- 수집 난이도: 중 (나라장터 낙찰 정보 분석 필요)

**제안**: Phase 1-2에서 수집 (입찰가 분석과 함께 구축)

---

### 🆕 **Category 9: 발주처 정보 & 특성 분석 (Client Intelligence)**

#### 9-1. 필요성
발주처별 제안 전략 수립에서:
- 이 발주처의 특성은?
- 평가 기준은 어떻게 되나?
- 과거 낙찰 사례로 본 경향은?

#### 9-2. 포함 내용
```
A. 발주처 기본 정보
  ├─ 발주처명, 담당부서
  ├─ 주력 사업분야
  ├─ 연간 공고 수
  └─ 과제 규모 경향

B. 발주처별 평가 특성
  ├─ 평균 기술점수 분포
  ├─ 평가 가중치 (기술 vs 가격)
  ├─ 낙찰률 (경쟁도)
  ├─ 재공고 빈도
  └─ "이 발주처는 기술 중시"

C. 발주처별 특성 분석
  ├─ 가격 민감도 (높음/중간/낮음)
  ├─ 품질 요구 수준
  ├─ 컴플라이언스 엄격도
  ├─ 사업 안정성 (재정/정치적 영향)
  └─ 평가위원 특성 (까다로움/관대함)
```

#### 9-3. 사용 방식
```
RFP 분석: "이 발주처와 과제의 특성"
  ↓
KB에서 발주처 정보 확인
  ↓
"국방부는 기술점수 편차 크고, 보안 중시"
  ↓
전략 수립: "기술력으로 차별화하고, 보안 강조"
```

#### 9-4. 유지관리
**우선순위**: **높음** (전략 수립 필수)
- 비용 대비 효과: 높음 (포지셔닝 결정에 영향)
- 수집 난이도: 중 (나라장터 + 내부 기록 분석)

**제안**: Phase 1에서 기본 정보만, Phase 2에서 심화 분석

---

### 🆕 **Category 10: 성과분석 & 입찰가 결정 (Bid Decision Analytics)**

#### 10-1. 현재 상태
사용자 확인: "입찰가 결정은 제안 워크플로우에 이미 설계되어 있음"
- 유사과제 낙찰가 분석 → 최적 입찰가 산정
- 낙찰가 데이터는 나라장터에서 자동 수집

#### 10-2. KB 역할
```
저장/관리할 데이터:
  ├─ 유사과제 낙찰가 DB (나라장터 수집)
  ├─ 포지셔닝별 승률 분석 (offensive vs defensive)
  ├─ 발주처별 낙찰가 범위 (평균, 최저, 최고)
  ├─ 입찰가 결정 기록 (실제 입찰가 vs 낙찰가)
  └─ 사후분석 (입찰가 적절도 평가)
```

#### 10-3. 사용 방식
```
입찰가 결정 단계:
  ↓
유사과제 낙찰가 5건 검색
  ↓
"50억 범위 과제의 평균 낙찰가: 46억 (92% 낙찰률)"
"Offensive 포지셔닝 시 낙찰률: 70%"
"Defensive 포지셔닝 시 낙찰률: 35%"
  ↓
의사결정 근거 제시
```

#### 10-4. 유지관리
**우선순위**: **최우선** (이미 구현되어 있음)
- 자동 수집 (나라장터 크롤링)
- 실제 낙찰 후 자동 기록
