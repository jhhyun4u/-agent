# Proposal Project Assets Lifecycle (제안 프로젝트 자산 관리)

**문제 정의**: 제안서 작성 과정에서 수집/생성되는 자료(공고서, RFP, 분석문서, 리서치 자료)들이 어디에 저장되고, 어떻게 활용되며, 작업 후 어떻게 관리되는지 명확하지 않음.

---

## 1. 제안 프로젝트 자료의 생명주기

### 1.1 Lifecycle 단계별 자료 분류

```
┌─────────────────────────────────────────────────────────────────────┐
│ 제안 프로젝트 자산 Lifecycle                                        │
└─────────────────────────────────────────────────────────────────────┘

Phase 0: 공고 발굴 (RFP Discovery)
  ├─ 공고서 (공고 공시 문서)
  ├─ RFP (Request for Proposal)
  ├─ 입찰 안내서 (나라장터, 조달청 규정)
  └─ 사전질의 응답 (있을 경우)

Phase 1: 분석 & 리서치 (Analysis & Research)
  ├─ RFP 분석 문서 (내부 분석 결과)
  ├─ 리서치 자료
  │  ├─ 특허 (관련 기술 특허)
  │  ├─ 논문 (학술 자료)
  │  ├─ 보고서 (정부 정책 보고서, 산업 보고서)
  │  ├─ 웹 자료 (뉴스, 기술 블로그)
  │  └─ 통계 (시장 통계, 벤치마크)
  ├─ 경쟁사 분석 (경쟁사 제안서 분석, 가격 예상)
  ├─ 발주처 분석 (발주처 성향, 과거 평가 배점 분석)
  └─ Go/No-Go 판정서

Phase 2: 전략 수립 (Strategy)
  ├─ 포지셔닝 문서
  ├─ Win Theme 정의
  ├─ 차별화 포인트 분석
  └─ 가격 책정 분석

Phase 3: 제안서 작성 (Proposal Writing)
  ├─ 제안서 구조안 (목차, 스토리라인)
  ├─ 섹션별 초안 (기술방안, 조직인력, 예산 등)
  ├─ 내부 리뷰 의견서
  ├─ 최종 제안서
  └─ 첨부 증빙 자료

Phase 4: 발표 준비 (Presentation)
  ├─ 발표자료 (PPT)
  ├─ 스토리보드
  ├─ Q&A 예상안
  └─ 발표 리허설 기록

Phase 5: 발표 & 평가 (Evaluation)
  ├─ 발표 기록
  ├─ 평가위원 질문
  ├─ 우리의 답변 기록
  └─ 평가 피드백 (기술점수, 가격점수, 의견)

Phase 6: 결과 & 폐기 (Post-Bid)
  ├─ 최종 점수 (낙찰/낙선)
  ├─ 사후 분석 보고서 (낙선 원인, 교훈)
  └─ 평가 결과 공시 자료
```

## 2. 현황 분석 (As-Is): 자료 관리 문제

### 2.1 현재 자료 저장의 문제점

| 단계 | 자료 | 현황 | 문제점 |
|------|------|------|--------|
| Phase 0 | 공고서, RFP | 각 팀이 다운로드 → 이메일, 폴더로 산재 | 중복 저장, 최신본 불확실 |
| Phase 1 | RFP 분석, 리서치 자료 | 개인 노트, 메모, 폴더 | 검색 불가, 재사용 어려움 |
| Phase 2 | 전략 문서 | 팀 공유폴더 또는 개별 | 누가 최신본인지 불명확 |
| Phase 3 | 제안서 초안, 의견서 | 버전 관리 없음 (proposal_v1_final_FINAL.docx) | 버전 추적 불가 |
| Phase 4 | 발표 자료 | PPT 파일로 산재 | 스토리보드와의 연결 불명확 |
| Phase 5 | Q&A 기록 | 팀원 노트 또는 메모 | 조직 차원 축적 안 됨 |
| Phase 6 | 평가 결과, 피드백 | 이메일, 문서로 전달 | 정리되지 않음, 학습 미흡 |

### 2.2 현재 시스템의 결과

```
결과 1: 자료 검색 불가
- "3년 전 의료 AI 제안에서 데이터 보안을 뭐라고 했더라?" → 찾기 어려움
- 시간 소비: 이메일 검색, 폴더 뒤지기 등

결과 2: 재사용 불가
- 유사 프로젝트의 분석 자료, 리서치 자료가 다시 사용 안 됨
- 매번 새로 리서치 해야 함 → 작성 시간 증가

결과 3: 학습 불가
- 낙선 원인, 평가 피드백이 정리되지 않음
- 같은 실수 반복 (예: "예산 설명 부족"으로 계속 낙선)

결과 4: 폐기 vs 보관의 모호함
- 어떤 자료는 폐기, 어떤 자료는 보관해야 하는지 기준 불명확
- 스토리지 낭비 또는 중요 자료 손실
```

## 3. 바람직한 구조 (To-Be): 제안 프로젝트 자산 관리 시스템

### 3.1 Category 8: 제안 프로젝트 자산 (Proposal Project Assets)

#### 3.1.1 저장 구조 (Storage Architecture)

```
제안 프로젝트 자산
│
├─ [공고/RFP 저장소] (Input Document Storage)
│  ├─ 공고 공시 문서 (원문)
│  ├─ RFP 전체 문서
│  ├─ 입찰 안내서 & 규정
│  ├─ 사전질의 응답
│  └─ 메타: 공고번호, 발주처, 공시일, RFP 접수 마감일
│
├─ [분석/리서치 저장소] (Analysis & Research Repository)
│  ├─ RFP 자동 분석 결과 (AI 분석)
│  │  └─ 요구사항 추출, 핵심 평가항목, 예상 가격대 등
│  │
│  ├─ 수동 분석 문서 (팀이 작성한 분석)
│  │  └─ Go/No-Go 판정서, 경쟁 분석, 발주처 분석
│  │
│  ├─ 리서치 자료 (관련 기술 및 정책 자료)
│  │  ├─ 특허 (원본 PDF + 요약 메모)
│  │  ├─ 논문 (원본 또는 링크 + 핵심 내용)
│  │  ├─ 보고서 (정부/산업 보고서 원본)
│  │  ├─ 웹 자료 (스크린샷 + 출처 URL)
│  │  └─ 시장 통계 (표로 정리된 데이터)
│  │
│  └─ 메타: 리서치 담당자, 수집일, 신뢰도, 출처
│
├─ [작업 문서 저장소] (Working Documents)
│  ├─ 제안서 구조안 (목차, 스토리라인)
│  ├─ 섹션별 초안 (버전 관리: v1 → v2 → v3 final)
│  ├─ 내부 리뷰 의견서 (PDF 또는 주석)
│  ├─ 전략 문서 (포지셔닝, Win Theme)
│  ├─ 가격 책정 분석
│  └─ 메타: 버전, 작성일, 작성자, 상태 (draft/review/final)
│
├─ [최종 산출물 저장소] (Final Deliverables)
│  ├─ 최종 제안서 (제출본)
│  ├─ 첨부 증빙 자료 (인력 경력서, 실적증명서 등)
│  ├─ 발표자료 (PPT + 스토리보드)
│  ├─ 발표 기록 (비디오/노트)
│  └─ 메타: 제출일, 수정 이력
│
└─ [평가 & 학습 저장소] (Evaluation & Learning)
   ├─ 평가 피드백
   │  ├─ 최종 점수 (기술/가격)
   │  ├─ 평가위원 의견
   │  ├─ 낙찰/낙선 판정
   │  └─ 공개 채점 결과 (있으면)
   │
   ├─ Q&A 기록
   │  ├─ 평가위원 질문
   │  ├─ 우리의 답변
   │  └─ 평가 반응
   │
   ├─ 사후 분석 보고서
   │  ├─ 낙선 원인 분석 (정성적)
   │  ├─ 경쟁 상황 분석 (낙찰사 정보)
   │  └─ 교훈 (다음에 개선할 점)
   │
   └─ 메타: 분석일, 분석자, 신뢰도
```

#### 3.1.2 저장 위치 (Where)

| 자료 유형 | 저장 위치 | 특성 | 접근성 |
|---------|---------|------|--------|
| 공고서/RFP | Supabase (문서 저장) | 원본 보존, 검색 필요 | 공개 또는 팀 전체 |
| 분석/리서치 | Supabase (메타 + 문서) | 재사용성 높음, 검색 필요 | 팀 접근 |
| 작업 문서 | Supabase + Git (버전 관리) | 버전 추적 중요 | 담당 팀 + 리뷰어 |
| 최종 산출물 | Supabase (아카이브) | 장기 보관, 법적 증거 | 제안팀 + 경영진 |
| 평가 & 피드백 | Supabase (메타 + 기록) | 학습용, 통계 분석 | 전사 (비식별화) |

#### 3.1.3 자동 정리 규칙 (Automation Rules)

**목표**: 제안 작업 완료 후 자동으로 분류, 태깅, 저장

```
Trigger: 제안서 제출 또는 평가 결과 확정

자동 작업:
1. 모든 작업 문서를 "최종 산출물 저장소"로 이동
   - 최종 제안서만 보관 (초안 제거 가능)
   - 리뷰 의견서 보관 (학습용)

2. 평가 피드백 자동 정리
   - 평가 점수, 의견 메타데이터로 저장
   - Q&A 분석 결과 Category 4 (Q&A 아카이브)로 연동

3. 리서치 자료 자동 분류
   - "이 리서치는 다음에 재사용 가능" 플래그
   - Category 4 (Q&A 아카이브)에 교훈 추가

4. 수행실적 업데이트
   - 낙찰 결과 → Category 2 (수행실적)에 추가
   - 낙선 → Category 7 (성과분석) 실패 원인 기록
```

## 4. 제안 프로젝트별 정보 아키텍처

### 4.1 "제안 프로젝트"의 메타 구조

```
Proposal Project Record
├─ 기본정보
│  ├─ Project ID (고유 번호)
│  ├─ 공고명 (RFP Title)
│  ├─ 발주처 (Client)
│  ├─ 분야/기술 (Field/Technology)
│  ├─ 계약금액 (Contract Value)
│  ├─ 공시일 (Posted Date)
│  ├─ 마감일 (Deadline)
│  ├─ 담당자 (Proposal Lead)
│  ├─ 팀 구성 (Team)
│  └─ 상태 (Status: 진행중/제출/결과대기/완료)
│
├─ 자료 연결
│  ├─ 입력 자료 (공고서, RFP)
│  │  └─ Document Links (Supabase)
│  │
│  ├─ 분석/리서치
│  │  ├─ RFP 분석 문서
│  │  ├─ 리서치 자료 (특허, 논문, 보고서)
│  │  ├─ 경쟁사 분석
│  │  └─ 발주처 분석
│  │
│  ├─ 제안서 작업
│  │  ├─ 전략 문서
│  │  ├─ 구조안
│  │  ├─ 제안서 초안 (버전 추적)
│  │  ├─ 최종 제안서
│  │  └─ 발표 자료
│  │
│  ├─ 평가 & 결과
│  │  ├─ 최종 점수
│  │  ├─ Q&A 기록
│  │  ├─ 피드백
│  │  └─ 사후분석 보고서
│  │
│  └─ 학습 연결
│     ├─ → Category 4: Q&A 아카이브 (Q&A 데이터)
│     ├─ → Category 2: 수행실적 (낙찰 시 추가)
│     └─ → Category 7: 성과분석 (점수, 포지셔닝, 승률)
│
└─ 타임라인
   ├─ Created (공고 발굴일)
   ├─ RFP Received (RFP 입수일)
   ├─ Analysis Complete (분석 완료일)
   ├─ Proposal Submitted (제출일)
   ├─ Presentation Date (발표일)
   ├─ Result Announced (결과 공시일)
   └─ Post-Analysis (사후분석 완료일)
```

└─ 리서치 자료
   ├─ 특허 5건: "분산 AI 모니터링" 관련 특허 분석
   ├─ 논문 3건: "비지도 이상탐지 알고리즘" 최신 연구

   └─ 시장 정보: 국방 IT 지출 통계

📝 제안서 작업
├─ 전략: 기존 AI 솔루션 기반 국방 특화 (보안 강화)

   ├─ PPT (20 슬라이드)

🎯 평가 & 결과
├─ 평가 점수
│  ├─ 기술점수: 85점 / 100점
│  ├─ 가격점수: 70점 / 100점
│  └─ 최종점수: 77.5점 (평균)
│
├─ Q&A 기록
│  ├─ Q: "보안 레벨 4를 어떻게 보증하나?"
│  │  └─ A: "국방 보안 인증 체계 준수 + 제3자 감시"
│  │
│  ├─ Q: "45억으로 충분한가?"
│  │  └─ A: "기존 솔루션 활용으로 개발 비용 절감"
│  │
│  └─ [기록: 평가위원 반응 → Category 4 연동]
│
├─ 결과
│  ├─ 낙선 (낙찰사: LG AI 연구소, 52억)
│  └─ 공개 채점 (확인됨)
│
└─ 사후분석
   ├─ 낙선 원인 분석
   │  ├─ 기술: "우리 보안 방안이 국방 표준과 맞지 않음"
   │  ├─ 가격: "45억은 적절했으나, 기술점수 차이로 뒤짐"

## 6. 정책 및 관리 규칙

### 6.1 자료 보관 정책 (Retention Policy)

| 자료 유형 | 보관 기간 | 폐기 기준 | 예시 |
|---------|---------|---------|------|
| 공고서/RFP | 3년 | 또는 법적 보존 기한 | 정부 계약 증명 목적 |
| 분석/리서치 | 3-5년 | 또는 유사 프로젝트 없음 | 재사용 가능한 동안 보관 |
| 작업 문서(초안) | 6개월 | 최종본 확정 후 폐기 | 최종본만 보관 |
| 최종 산출물 | 5년+ | 법적 증거 필요 시까지 | 완전 보관 |
| 평가 피드백 | 장기 (무제한) | 통계 분석 완료 후 비식별화 | 조직 학습용 |
| 리서치 자료 | 3년 | 또는 기술 노후 판단 | 시장/기술 변화 반영 |

### 6.2 접근 제어 (Access Control)

```
공고서/RFP:        팀 전체 (읽기) + 제안 리드 (쓰기)
분석/리서치:        제안팀 (읽기) + 담당자 (쓰기)
작업 문서:         담당팀 (읽기/쓰기) + 리뷰어 (읽기)
최종 산출물:        팀 전체 (읽기) + 제안 리드 (관리)
평가 피드백:        리더십 + 제안팀 (읽기) + 입력자 (쓰기, 비식별화)
```

### 6.3 명명 규칙 (Naming Convention)

```
공고서:           PROP-{YYYY}-{MM}-{공고번호}.pdf
분석문서:         ANALYSIS-{Project ID}-{문서유형}-{v버전}.docx
제안서:           PROPOSAL-{Project ID}-{상태}-{YYYYMMDD}.docx
  예시:
  - PROPOSAL-PROP-2026-0045-DRAFT-20260401.docx
  - PROPOSAL-PROP-2026-0045-FINAL-20260408.docx
  - PROPOSAL-PROP-2026-0045-SUBMITTED-20260410.docx
  
발표자료:         PRESENTATION-{Project ID}-{YYYYMMDD}.pptx
평가결과:         EVAL-RESULT-{Project ID}-{공시일자}.json
```

## 7. 새로운 질문: Category 추가 여부

### 옵션 A: Category 8 신규 추가
**"Proposal Project Assets (제안 프로젝트 자산)"**

포함 항목:
- 공고서/RFP 원본
- 분석/리서치 자료
- 작업 문서 (버전 관리)
- 최종 산출물
- 평가 결과 & 피드백

장점:
- 제안 프로젝트의 **전체 생명주기** 관리
- 자료 검색 + 재사용성 극대화
- 자동 학습 루프 구축 (평가 결과 → 다음 제안에 반영)

단점:
- 복잡도 증가
- 스토리지 증가 (모든 버전 추적)

### 옵션 B: 기존 7 Category와 통합
**Category 3, 4, 6, 7에 분산 관리**

- Category 3 (템플릿): 제안서 최종본 + 섹션별 우수 예시
- Category 4 (Q&A): 평가 질문 + 우리 답변 자동 연동

장점:
- 기존 구조 유지
- 복잡도 낮음

단점:
- 제안 프로젝트의 **원본 자료**(공고서, RFP, 리서치)가 관리 안 됨
- 버전 추적 불가
- 자료 검색 어려움
